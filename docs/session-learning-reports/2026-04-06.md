  - `.omx/plans/prd-phase-4-performance-truthfulness.md` (APPROVED, but
    implementation shipped 5 hours before the architect-review artifact was

  - `docs/plans/2026-04-05-todo-report-remediation-strategy.md` (Worktracks
    A/B/C1 all shipped in `6bb2a92d`, `dabc93be`, `72c7d834` without doc
    updates)
- **Category:** Process / Workflow
- **Anti-Pattern:** Treating a planning doc's `status:` frontmatter as
  authoritative about what work is open. The doc represents a point-in-time
  snapshot; commits can land faster than plans get reconciled.
- **Observation:** Three distinct instances of this pattern in a single session.
  Each cost meaningful investigation time.
- **Fix:** Before scoping any task from a planning doc, verify each claim
  against current `main` via grep/git log. Evidence commands beside each claim
  (which the todo-report remediation strategy doc does correctly) make this
  trivial.
- **Recommendation:** CREATE FEEDBACK MEMORY (user-personal)

### Candidate 2: Parallel sessions need planning-doc reconciliation FIRST

- **Score:** 3 (repeated pattern +1, DX friction +1, risk surface +1)
- **Source:** `.omx/context/time-hardening-20260406T081749Z.md` + conversation
  around Phase 4 close-out plan dispatch
- **Category:** Process / Workflow
- **Anti-Pattern:** Running two concurrent sessions on the same repo where one
  is doing planning research while another is about to execute code changes. The
  planning session reads stale PRDs and draws wrong conclusions about what's
  in-flight; the execution session may collide at the file level if the planned
  lane overlaps.
- **Fix:** When parallel sessions are expected, the first action on the
  execution branch should be planning-doc frontmatter reconciliation — mark
  COMPLETE/SUPERSEDED/ARCHIVED on anything already shipped. This takes minutes
  and unblocks the planning session with truthful ground-state. The code changes
  can wait.
- **Recommendation:** CREATE FEEDBACK MEMORY (user-personal) — fold into
  Candidate 1 as a second rule.

### Candidate 3: Dependabot dashboard lies about severity until lockfile rescans

- **Score:** 4 (security +2, DX friction +1, repeated pattern +1)
- **Source:** dependabot API response + `npm audit` comparison during the sweep
- **Category:** Infrastructure / Security
- **Anti-Pattern:** Taking the dependabot banner count ("21 vulnerabilities: 9
  high") at face value when planning remediation work. GitHub's scanner is
  asynchronous — if the lockfile already has a patched version but no new commit
  has been pushed, the dashboard keeps showing stale alerts indefinitely.
- **Observation:** In this session, **9 of 21 alerts were already-patched false
  positives** (rollup, picomatch, postcss, minimatch v10, vite). Another **4
  were scanner false positives against the wrong version range** (vite v7
  advisory applied to our v5). Only **8 alerts represented real actionable
  work** — and even those split into real code fixes (5) and dev-only
  accepted-risk transitives (3).
- **Fix Sequence:**
  1. Cross-reference every open dependabot alert against the current lockfile
     _before_ planning any remediation.
  2. Ship real patches first; pushing the updated lockfile triggers GitHub's
     rescan which auto-clears many stale false positives.
  3. For alerts that won't auto-clear (either because the vulnerability is real
     or the scanner is stuck), dismiss with evidence via
     `gh api --method PATCH repos/:owner/:repo/dependabot/alerts/:n -f state=dismissed -f dismissed_reason=<reason>`.
  4. Valid dismissal reasons: `fix_started`, `inaccurate`, `no_bandwidth`,
     `not_used`, `tolerable_risk`. `dismissed_comment` is limited to **280
     characters** (Twitter-length, not documented in obvious places).
- **Recommendation:** CREATE REFERENCE MEMORY (user-personal)

### Candidate 4: Nested overrides for parent-scoped transitive fixes

- **Score:** 2 (DX friction +1, reusable technique +1)
- **Source:** `package.json` override added in commit `0298a65c` to force
  `path-to-regexp@0.1.13` only under `@lhci/cli` without touching the global
  `path-to-regexp@8.4.0`
- **Category:** Build / Dependencies
- **Technique:**
  ```json
  "overrides": {
    "@lhci/cli": {
      "path-to-regexp": "^0.1.13"
    }
  }
  ```
  Forces the transitive version _only_ under a specific parent, preserving the
  higher version elsewhere in the tree. Useful when a legacy dev tool carries an
  old transitive but other packages rely on a newer major.
- **Recommendation:** FOLD INTO Candidate 3 reference memory

### Candidate 5: Stale session-start git status

- **Score:** 2 (DX friction +2)

- **Category:** Git / Tooling

- **Fix:** Re-run `git status` / `git log` immediately before any git action,
  especially when the initial state was captured minutes or hours earlier.
