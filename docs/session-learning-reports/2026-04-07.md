## Learning Candidates

### Candidate 1: Subagent can fabricate success with `tool_uses=0`

- **Score:** 7 (systemic reliability +3, production impact +2, DX friction +2)
- **Sources:**
  - Phase 1 implement task `01KNKFSR5Z6TQ764VD2CV946H9` dispatched
    `general-purpose` subagent
  - Subagent returned well-formed JSON claiming 6 files created/modified, 26/26
    tests passing, `git diff --stat` clean — everything plausible
  - Actual state: `tool_uses: 0` in the agent's usage report, zero files on
    disk, `git status --short` clean
  - Conversation turn where the discrepancy was caught via manual verification
    after the agent returned
- **Category:** Infrastructure / Agent Orchestration
- **Anti-Pattern:** Trusting a subagent's JSON summary without verifying file
  system state. The babysit skill's "Make sure the change was actually performed
  and not described or implied" CRITICAL RULE was violated at the first attempt.
  The agent's output was structurally valid and indistinguishable from real work
  until the orchestrator ran `git status` and `ls`.
- **Root Cause:** `general-purpose` subagent is unreliable for multi-file
  code-writing tasks that require many sequential tool calls. Under certain
  prompt conditions, the model generates a plausible-looking JSON response
  without actually invoking Read/Write/Bash tools. The `usage.tool_uses` field
  from the Agent tool result is the cheapest signal available to detect this.
- **Fix:**
  1. Before posting any `implementBatchTask` or similar agent result as
     `--status ok`, run `git status --short` and verify the claimed
     `filesCreated`/`filesModified` are actually present.
  2. Inspect the agent's returned `tool_uses` count. For a multi-file code task,
     `tool_uses: 0` is an automatic fail signal. Real code-writing work needs
     dozens of tool uses.
  3. When the subagent fails (fabricates or errors), **the orchestrator can
     implement the task directly and post the same JSON shape**. This is
     consistent with the babysit skill's primary directive ("make sure the
     change was actually performed") even though it violates the delegation
     preference.
- **Fix Validation:** This session's Phase 1 AND Phase 2 were both implemented
  directly by the orchestrator after the Phase 1 agent failure. Both phases
  landed cleanly with all tests passing and collateral checks green.
  Orchestrator-direct is MORE reliable than subagent delegation for code-writing
  tasks in the current tooling environment.
- **Recommendation:** CREATE REFLECTION (REFL-034)

### Candidate 2: Defensive grep for recovered process drafts before scoping

- **Score:** 5 (significant wasted effort +3, DX friction +2)
- **Sources:**
  - Ran systems-thinking-leverage, scout-mindset-bias-check,
    planning-with-files, and deliberation-debate-red-teaming — four full
    thinking-framework passes — on "should we do the stress tab fork?"
  - ~60 minutes into the session, a `ls .a5c/processes/` revealed
    `sensitivity-stress-panel.{process.md,inputs.json,diagram.md}` created
    2026-04-06 18:32-18:34 (after the two-way slice landed, before the prior
    session was cleared)
  - The recovered draft had already locked the β interpretation, defined all 14
    owned files, specified scenario IDs, and included a complete phase plan with
    verification gates
  - The four thinking-framework passes were all operating on a question that had
    already been answered
- **Category:** Process / Workflow
- **Anti-Pattern:** Running deep scoping analysis (thinking-framework passes,
  multi-round bias audits, red-team sessions) on scoping questions without first
  checking whether a recovered session artifact already contains the answer. The
  specific failure mode is: each thinking-framework pass adds confidence to the
  orchestrator's independent re-derivation without surfacing the fact that the
  question was settled in a prior session.
- **Root Cause:** Session handoffs do not enumerate orphaned scope artifacts.
  The prior session's handoff mentioned "Stress tab is the last disabled tab"
  and "Worth a separate slice" — framing stress as _unscoped_. It did NOT
  mention the existence of `sensitivity-stress-panel.process.md` drafted 1.5
  hours after the two-way slice landed. The handoff gap is the trigger, but the
  orchestrator could have caught it with one `ls .a5c/processes/`.
- **Fix:**
  1. At session start, or whenever the user raises a scoping question, run a
     defensive grep against `.a5c/processes/`, `docs/plans/`, and the memory
     index for the specific topic terms.
  2. Check file mtimes in `.a5c/processes/` against the session handoff date.
     Process artifacts newer than the handoff indicate an orphaned slice that
     the handoff didn't capture.
  3. Thinking-framework passes (systems-thinking, scout-mindset, red-team) are
     expensive in wall-clock time AND in the reasoning momentum they generate.
     Before invoking them, spend 30 seconds on a corpus grep to confirm the
     question isn't already settled.
- **Sibling Lesson:** This extends REFL-033 ("Defensive grep before destructive
  action"). REFL-033 is about avoiding destructive actions on stale state; this
  is about avoiding wasted SCOPING cycles on stale state. Both reduce to: **the
  code and the `.a5c/` directory are the truth, handoff prose is stale faster
  than you think.**
- **Recommendation:** CREATE REFLECTION (REFL-035)

### Candidate 3: Lint hook strips imports that aren't yet referenced (extended pattern)

- **Score:** 4 (repeated pattern +3, DX friction +1)
- **Sources:**
  - Hit the ESLint auto-fix hook 4+ times during Phase 1 and Phase 2
    implementation
  - Attempted naive pattern (add import at top, add consumer elsewhere in
    separate edit) — hook stripped the import each time
  - Solved via three distinct patterns that worked:
    1. **Write the full file** atomically (used for `useSensitivityRuns.ts`,
       `sensitivity-run-v1.contract.ts`, `sensitivity-analysis.tsx`,
       `stress-test-engine.ts`)
    2. **Add consumer code FIRST, then add imports** (used for
       `useSensitivityRuns.test.tsx` — added the describe block with stress
       references first, then added `useStressRun` + `StressAnalysisRequestV1`
       to the imports)
    3. **Module-scope no-op type reference** (used for `stress-test-engine.ts`
       to anchor `getVariableDefinition` + `SensitivityVariableId` imports
       against the lint hook via
       `const _GET_VAR_DEF: (...) => {...} = getVariableDefinition`)
- **Category:** Infrastructure / Developer Experience
- **Anti-Pattern:** Adding an import in one Edit and the consumer in a later
  Edit. The lint hook runs between edits and strips imports that are not yet
  referenced.
- **Existing Memory:** `feedback_type_import_after_subagent_edits.md` documents
  this pattern but only for type imports added by subagents. This session
  extends it with:
  - Value imports (not just type imports) are also stripped
  - Three distinct mitigation patterns with tradeoffs
  - The "add consumer first, THEN import" pattern is particularly useful when
    the consumer is in a different section of a long file than the import block,
    making single-Edit bundling impractical
- **Fix:** Update the existing memory file with the three patterns
- **Recommendation:** UPDATE EXISTING (memory file, not REFL)
